Brief Description of System

Es un conjunto de agentes/servicios capaces de planificar una visita turistico-cultural a una ciudad en sentido amplio. Esta planificacion se adaptara a las preferencias y restricciones que llegan por parte de un agente externo que representa a un usuario. El sistema consta de servicios que filtran/obtienen/seleccionan toda la informacion necesaria, los servicios que generan esa planificacion y todos aquellos servicios que son adicionales, que permiten gestionar todos los elementos que forman parte de una solucion. Por otra parte, el sistema emplea servicios que favorecen el correcto funcionamiento del mismo



System Specification (Primera Fase)

Analysis Overview Diagram

Para la realizacion de este diagrama hemos hecho un analisis global del sistema, empezando de forma cronologica con la primera interaccion del Usuario (mediante el cliente web).

Asi pues, tenemos que el sistema se va a lanzar cuando reciba una percepcion del cliente web pidiendole un plan de viaje. Entonces, el sistema dara comienzo a la busqueda de los datos necesarios (transporte, alojamientos y actividades) segun las exigencias recibidas por el usuario a traves de sus respectivos Actores.

Una vez recabados los datos, el sistema recibe las percepciones Datos de Actividades, Datos de Alojamiento y Datos de Transporte, y monta el plan de viaje e informa del plan al usuario, para que sopese si le interesa o no.

Si el plan es aceptado, se procede a realizar el pago mediante el actor Admin. de Pago. Este, genera una percepcion que recoge el sistema y acciona la comunicacion con el usuario, informandole de que el pago se ha realizado.

Una vez el usuario ha terminado el viaje, dado que el sistema tiene almacenadas las fechas del plan, se le pide una valoracion al usuario para ser tratada de forma proactiva.



Goal Overview Diagram

En esta visualizacion dejamos explicito el mapa de objetivos que hemos extraido de las funcionalidades que llevara a cabo nuestro sistema.

Constatamos 4 bloques interrelacionados entre ellos:

  1. Bloque Gestion del Plan de Viaje: debera cumplir con la obtencion de datos; tanto sobre los medios de transporte, como los alojamientos o las actividades. Luego, esos datos requeriran ser filtrados en una Criba de opciones para Organizar el Plan, y finalmente, se informara de dicho plan para que el usuario pueda tomar una decision.
  2. Bloque Gestion de Pago, en el que para cumplirlo, primero es necesario realizar el pago -es decir, obtener los datos de pago y mandar propiamente a pagar- e informar del pago.
  3. Bloque Actualizar datos. Para este objetivo, iremos obteniendo y escribiendo de forma periodica los datos externos en nuestra base de datos.
  4. En el bloque de Autenticacion, se cumplira con el objetivo de registrar y con el de identificar o LogIn.

System Roles Diagram

En el siguiente diagrama vemos representados los roles del sistema, con sus percepciones y acciones que nos ayudan a cumplir todos los objetivos -de sus respectivos escenarios- descritos hasta el momento.

De todos estos, por ser quizas el mas complejo, parece merecer la pena hacer mencion al rol Comunicador. Dicho rol, es el encargado de mantener siempre una via de interaccion y comunicacion con el usuario en el transcurso de todo el proceso. Eso es: desde el momento en el que recibe su primera percepcion de peticion de plan, donde se informara del plan y, si acepta, del pago; hasta el momento de obtener la valoracion del usuario sobre el viaje, una vez este haya pasado. Tambien recogera informacion de las preferencias del usuario.



Architectural Design (Segunda Fase)

Agent-role Grouping Diagram

Con este esquema identificamos cuales van a ser los agentes de nuestro sistema, y hacemos un mapeo grafico con los roles -deducidos anteriormente- que van a ejercer cada uno de ellos.

Cabe destacar la decision tomada en juntar finalmente los roles de obtencion de alojamiento y actividades en un mismo agente, ya que pensamos aprovechar ciertas cualidades de connivencia entre el lugar donde se hospeda el turista y las actividades que pueden resultarle de interes (p.e: proximidad, movilidad, etc.).



System Overview Diagram

En este apartado hemos hecho el diagrama en que ya asociamos todas las percepciones y acciones con sus respectivos agentes, identificando tambien que bases de datos van a acceder cada uno de ellos y, si van a necesitar comunicarse, especificamos el protocolo de comunicacion que van a usar.

Los protocolos se mencionan en el siguiente apartado, y hemos definido y especificado hasta ahora todas las percepciones y acciones del sistema; asi que nos vamos a centrar en la comparticion de fuentes de datos.

Para la fuente de datos de planes, el agente pagador actualizara los planes para indicar que ya ha sido satisfactoriamente pagado, mientras que el planificador escribira todos esos planes que han sido aceptados por el usuario.

Otra fuente compartida es la de datos de transporte, entre el gestor de transporte y el actualizador de datos. Ambos escribiran y leeran de esa base de datos; el primero para recuperar los datos, que deben ser comunicados al agente planificador, y el segundo, debera actualizar los datos en funcion de las busquedas mas frecuentes.

De forma totalmente independiente tendremos el agente autentificador con una base de datos de usuarios para el, desde donde registra y lee las credenciales de los usuarios.



Protocols


Entre el Agente Administrativo y el Agente Pagador habra una comunicacion bidireccional, en el que el primero le hara una peticion de pago y el segundo le transmitira el informe resultante.

Entre los otros dos, habra tambien un protocolo de comunicacion bidireccional en el que el Agente Planificador les pedira y recibira datos de los Agentes de Alojamiento y Actividad asi como el de Transporte.

El mensaje, evidentemente, sera un conjunto de transportes, alojamientos o actividades que cumplen con las exigencias expuestas que le han llegado al agente planificador.

Los protocolos que intervienen en el sistema son:

  • Protocol Pr. de pago
  • Protocol Pr. Datos Aloj. y Act.
  • Protocol Pr. Datos Transporte
  • Protocol Pr. Plan creado
